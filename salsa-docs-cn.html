Launching payroll with Salsa Express · 用 Salsa Express 上线薪资

本指南从宏观层面概述如何使用 Salsa Express 上线一个薪资产品。若需要更详细的分步教程,请访问 Build a Payroll App with Salsa Express 页面。

Salsa Express 集成是把一个稳健、合规、设计精美的薪资产品嵌入你平台的最快路径。只需一名工程师、四周时间,你就能借助 Salsa 的可嵌入组件、灵活的 API 和实时 webhook,上线一套功能齐全、贴合你品牌调性的薪资体验。

🚀 从零到薪资的 3 个步骤

  1. 在你的应用内嵌入我们的组件
  2. 与 Paystream API 集成以导入数据
  3. 监听我们推荐的 webhook 事件

👍 核心收益

  • 速度:一名工程师,4 周内上线。
  • 掌控力:把雇主数据和薪酬数据保留在你自己的平台内。
  • 体验:用可嵌入的 UI 取悦用户,给他们原生般的感受。
  • 合规:所有薪资、税务和申报的复杂性都由 Salsa 处理。

Salsa 的 Paystream API 让你能导入你已经拥有的数据,而我们的可嵌入组件让你能提供世界级的薪资工作流,并具有高度可定制性——从样式到结构皆可定制。你可以调整它们以匹配你品牌的外观与感觉,并有选择地启用或禁用功能。此外,我们的 webhook 通知会在重要事件发生时告知你,让你用自己的通知系统来交付客户通知和薪资文档。

1. 在你的应用内嵌入我们的组件

为了嵌入我们的组件,你只需遵循 3 个简单步骤:

获取 API Token

要获取我们沙盒(sandbox)环境的 API token,你只需与我们联系,我们很快就会向你提供一个。

  • Salsa 要求在所有发往我们 API 的请求的 Authorization 头中提供一个 OAuth 2.0 Bearer Token。
  • User Token 让你能为某个用户临时授予访问权限,并用一个访问角色(access role)来指定其对你数据的访问级别。生成的 token 具有可配置的过期时间。

导入我们的 Salsa.js 库

加载并初始化 Salsa.js 库可简化为以下代码。为了把组件定制成你自己的品牌样式,你只需要一个 Client Key,我们会向你提供。

import { useScript } from './hooks/useScript';

useScript('https://js.salsa.dev/v0');
const [salsa, setSalsa] = useState();
 if(window.Salsa && !salsa) {
    setSalsa(window.Salsa('my-public-client-key', {env: 'sandbox'}));
 }
// Load Salsa.js into your application
<script type="text/javascript" src="https://js.salsa.dev/v0" />

// Create an instance of Salsa.js using your Client Key and specifying the environment
const salsa = window.Salsa('my-public-client-key', {env: 'sandbox'});

利用 Salsa.js 来嵌入我们的组件

你只需实现以下几个可嵌入组件,就能快速启动薪资功能,之后再通过实现更多组件来扩展:

  • Employer Onboarding(雇主入职):把薪资所需的全部设置汇集在单一体验中。
  • Employer Profile(雇主资料):用于查看和编辑雇主的设置信息。
  • Employer Dashboard(雇主仪表盘):供雇主运行薪资的管理面板。
  • Worker Profile(员工资料):用于管理员工的设置信息。
  • Worker Payments List(员工付款列表):为员工提供其付款的列表。

为嵌入元素创建容器,并把元素挂载(mount)到容器中

<div id='salsa-container' />

const salsaElement = salsa.elements.create('demo');
salsa.elements.mount(salsaElement, '#salsa-container');

ℹ️ 我们还以托管体验(hosted experiences)的形式提供 Employer Onboarding 和 Worker Onboarding。可以通过 API 取得一个邀请 URL,并把它发送给雇主和/或员工,供他们访问这些体验。

2. 与 Paystream API 集成以导入数据

Paystream 是我们推荐用来把任何数据从你的平台导入 Salsa 的 API。它有多个端点,允许你导入雇主(employers)、员工(workers)或薪酬条目(pay entries)。

导入雇主(Importing Employers)

一旦某个雇主在你的平台上注册了薪资服务,你就可以使用 Paystream - Import Employers 端点把雇主数据导入 Salsa。

curl --location 'https://api.salsa.dev/api/rest/v1/paystream/employers' \
--header 'Content-Type: application/json' \
--header 'Authorization: Bearer <auth_token>' \
--data '{
    "data": [
        {
            "type": "PaystreamEmployerUpsertInput",
            "externalId": "myplatform_123",
            "businessInfo": { "businessName": "Beauty Hair Salon" }
        }
    ]
}'

导入员工(Importing Workers)

如果你的平台拥有某个雇主旗下员工的数据,你可以使用批量导入端点 Paystream - Import Workers 来导入他们。

curl --location 'https://api.salsa.dev/api/rest/v1/paystream/workers' \
--header 'Content-Type: application/json' \
--header 'Authorization: Bearer <auth_token>' \
--data '{
    "data": [
        { "type": "PaystreamWorkerUpsertInput", "employerId": "<example_employer_id>", "firstName": "John", "lastName": "Doe" },
        { "type": "PaystreamWorkerUpsertInput", "employerId": "<example_employer_id>", "firstName": "Mary", "lastName": "Smith" }
    ]
}'

导入薪酬条目(Importing pay entries)

如果你的平台拥有员工薪酬数据(例如工作小时数或薪资费率),可以发送过来自动填充一份薪资单(payroll),为客户省下数小时手动录入。使用 Paystream - Import payroll elements。

curl --location 'https://api.salsa.dev/api/rest/v1/paystream' \
--header 'Content-Type: application/json' \
--header 'Authorization: Bearer <auth_token>' \
--data '{
    "data": [
        { "type": "PaystreamPayEntryUpsertInput", "workerId": "<id_1>", "payReferenceId": "ref:pay:saasexample:us:salary", "startDate": "2025-01-01", "endDate": "2025-01-31", "amount": "3000.00" },
        { "type": "PaystreamPayEntryUpsertInput", "workerId": "<id_2>", "payReferenceId": "ref:pay:saasexample:us:hourly", "startDate": "2025-01-03", "endDate": "2025-01-03", "hours": "8", "rate": "25" }
    ]
}'

3. 监听我们推荐的 webhook 事件

为了开始运行你的薪资应用,我们提供了一组 webhook 事件,它们可以在你的运营侧、或需要与 Salsa 同步数据时提供帮助。建议先从监听以下事件开始:

  • PayrollRun.awaitingInput:当某次薪资运行(payroll run)需要薪酬数据时触发。收到此事件时,你必须在所提供的时间区间内,通过 Paystream 把所列员工的全部薪酬条目(pay entry)数据发送过来。如果你的产品对于何时发送薪酬数据已有清晰流程,则此事件非必需。
  • Worker.onboardingStatus:每当某个员工被邀请入职、或已开始/完成入职时触发。
  • Notification.employerPaydayReminder:在雇主必须确认下一次已排期的薪资运行之前 2 天触发。
  • Notification.employerPayrollRunConfirmation:当某个雇主确认一次薪资运行时触发。
  • Notification.workerPayday:在员工被支付薪资的当天触发(当天 UTC 下午 1 点或晚上 11 点发送)。

Authorization · 授权(鉴权)

安全地访问 Salsa 提供的所有数据和能力。

对我们 API 发起的所有请求,Salsa 都要求在 Authorization header 中携带一个 OAuth 2.0 Bearer Token。支持两种 token 类型:API Tokens 和 User Tokens。

API Token

Salsa 提供两个 API Token 用于访问我们的 API:一个用于 Sandbox 环境,一个用于 Production 环境。这些 token 授予了很多权限,务必妥善保管,切勿公开使用或分享。

向 Salsa API 发起请求时,请使用以下 base URL:

  • Sandbox: api.sandbox.salsa.dev
  • Production: api.salsa.dev

找不到你的 API token? 如果弄丢了 token 或需要重新发送,请联系我们。

User Token

User Token 让你可以临时为某个用户开通访问权限。通过在创建时指定一个 access role(访问角色),你可以指定该用户对你数据的访问级别。

要创建一个 User Token,向 Salsa 的 Credentials API 发送请求。下例演示在 sandbox 环境中,以 EMPLOYER_ADMIN 角色创建 User Token:

curl --request POST \
     --url https://api.sandbox.salsa.dev/api/rest/v1/auth/token \
     --header 'accept: application/json' \
     --header 'authorization: Bearer ${YOUR_SANDBOX_API_TOKEN}' \
     --header 'content-type: application/json' \
     --data '{
          "type": "CreateEmployerUserTokenInput",
          "role": "EMPLOYER_ADMIN",
          "employerIds": ["${SALSA_EMPLOYER_ID_1}", "${SALSA_EMPLOYER_ID_2}"]
        }'

该例子中生成的 token 有效期为 1 小时,过期后需签发新 token。过期时间可配置。

Available Roles(可用角色)

角色分三个层级:basic、admin 和 super-admin。每一层级都授予其下一层级的所有权限,并追加更多权限。每个层级还有 *_ONBOARDING_* 变体,只拥有 UI Experiences 的 onboarding workflows 所必需的、经过缩减的权限。用户可被启用以临时提升(step up)到更高角色。

Role说明
EMPLOYER_BASIC一组基本权限。无法执行敏感操作(如添加银行账户)。
EMPLOYER_ONBOARDING_BASIC同 EMPLOYER_BASIC,但只拥有 onboarding workflows 所需权限。
WORKER_BASIC一组基本权限。无法执行敏感操作。
WORKER_ONBOARDING_BASIC同 WORKER_BASIC,但用于 onboarding workflows。
EMPLOYER_ADMIN查看和修改所有与 employer 关联的数据(workers、payroll runs 等)。
EMPLOYER_ONBOARDING_ADMIN同 EMPLOYER_ADMIN,但只限 onboarding。
WORKER_ADMIN查看和修改所有与 worker 关联的数据(个人信息、payment records 等)。
WORKER_ONBOARDING_ADMIN同 WORKER_ADMIN,但只限 onboarding。
EMPLOYER_SUPER_ADMIN同对应 admin,再加特权敏感操作,如解除掩码显示完整银行账号和政府证件号(SSN/TIN)。
EMPLOYER_ONBOARDING_SUPER_ADMIN同上,但只限 onboarding。
WORKER_SUPER_ADMIN同对应 admin,再加特权敏感操作(解除掩码显示 SSN/TIN 等)。
WORKER_ONBOARDING_SUPER_ADMIN同上,但只限 onboarding。

Sensitive and privileged actions(敏感操作与特权操作)

敏感操作按层级门禁。添加银行账户或编辑 worker 政府证件号的值需要 admin。解除掩码显示完整银行账号或政府证件号、移除 worker 银行账户、下载含政府证件号的 Worker Details 报表,都需要 super-admin。basic 角色都不能做。

worker 政府证件号的要求(是 MANDATORY 强制,还是仍处于 APPLIED_FOR 已申请状态)属普通 worker 数据,非敏感数据,任何能编辑该 worker 的角色都可设置。

这些限制在 API 层面强制,而非仅 UI。若想按需授予更高层级操作而不直接签发更高角色,请用 step-up(权限提升)流程。

Build with Salsa · 用 Salsa 来搭建

学习使用 Salsa 的 APIs、UI 组件和 Webhooks 来搭建薪酬(payroll)解决方案。

我们的目标,是把薪酬的种种复杂之处简化,让你能专注于打造出色的产品。我们提供 APIs、可嵌入的 UI 组件,以及 Webhook 通知,让你快速部署自己的薪酬体验;与此同时,由我们的后台(back office)处理所有报税与缴税事务,把你从应对联邦、州和地方各级机构的种种难题中解放出来。

Typical flow of funds · 典型的资金流转

Salsa 配备了用于向工人(workers)和第三方发放资金的资金流转通道(money movement rails)。我们负责欺诈监控、了解你的业务(Know-Your-Business,KYB)流程,以及反洗钱(Anti-Money Laundering,AML)合规;你只需输入数据,即可完成对工人的付款。

API rate limits · API 调用频率限制

Salsa 会限制单个 IP 或单个 API token 可以发起的请求数量,以防止对我们平台的恶意或意外滥用。如果你的正当使用量开始接近这些限制,我们可以调整它们以适配你的需求。

IP rate limit · IP 频率限制

同一个 IP 地址每 5 分钟最多可发送 3000 个请求。

API Token rate limit · API Token 频率限制

拥有 API token 的用户每 5 分钟最多可发送 3000 个请求,无论 IP 地址是什么。

Rate limited Response · 被限流的响应

当达到或超过频率限制时,Salsa 返回 429 - Too Many Requests HTTP 状态码。

Deep Dives 深入专题

Employer sign-up workflow · 雇主注册流程

了解并跟踪雇主 payroll 注册流程的每一个阶段。

处理 payroll 意味着要和大量数据打交道,尤其是在雇主 onboarding 阶段。我们设计了一套高效的注册流程,把你的数据与 Salsa 无缝集成。本指南使用 Salsa Express 组件来展示这个注册流程。

Payroll setup · Payroll 设置

Salsa 通过收集关键数据、执行合规检查、以及从此前的服务商无缝迁移 payroll 总额,来简化给 worker 发薪。整个流程分三个核心阶段:onboarding(入驻)、implementation(实施)和 employer account verification(雇主账户验证)。

  • Onboarding(入驻):初始阶段收集来自雇主的必要税务和财务信息,为第一次 payroll run 做准备。流程设计得直接,对大多数雇主都熟悉易懂。
  • Implementation(实施):对于在同一财政年度内更换 payroll 服务商的雇主至关重要。它涉及导入此前的 payroll 总额,以确保税务记录和缴纳准确,并保持 worker 的年初至今(year-to-date)收入一致。
  • Employer Account Verification(雇主账户验证):在美国和加拿大等地区,payroll 公司必须遵守特定合规标准。这涉及对雇主和 worker 进行强制筛查,以满足 KYB(了解你的业务)和 AML(反洗钱)法规。多数情况下检查会无缝执行。

Monitoring · 监控

注册过程中,监控 Employer Pay Worker Readiness(雇主给 worker 发薪的就绪度)和 Employer Account Status(雇主账户状态)很重要。前者确保雇主已提供发薪所需全部数据,后者确认雇主满足 KYB 和风险要求可在平台运营。两个字段都可通过 Dashboard 或 API 访问。

二者独立,因为注册流程并不总是按顺序进行:

  • Employer onboarding 和 implementation 是直接、连续的步骤,onboarding 只发生一次。
  • Account verification 通常在 implementation 之前结束,运作更独立。若需延伸验证,你的客户支持团队会与 Salsa 协作获取额外细节。

📘 Continuous risk monitoring · 持续的风险监控

注册之后,雇主会被持续监控风险和 KYB,这可能改变其 Account Standing(账户信誉状况),且与其发薪数据要求相互独立。

Worker payment returns · 员工付款退回

当一个薪资运行(payroll run)被确认后,Salsa 开始将计算出的应付金额从雇主银行账户转移到员工银行账户。这个资金流转包含两笔主要的 ACH 交易:

  1. Employer Funding(ACH debit)——从雇主银行账户提取资金。
  2. Payouts(ACH credits)——将资金推送到员工银行账户。

ACH 交易处理通常有前置时间(lead time)。为确保及时向员工付款,payouts 会在 employer funding 完全处理完成之前的 2 到 4 个工作日就发起。

How it works · 运作方式

员工付款可能因各种原因失败,例如雇主账户资金不足,或银行账户被冻结。发生时,Salsa 用以下流程管理被退回的付款:

  • 自动将 employer payouts 及相关税款转账暂时置于保留(on hold),确保资金问题解决前不处理后续交易。
  • 尝试用 payment return codes 识别并归类失败根因,然后纠正情况并成功重新扣款(re-debit)雇主账户。
  • 向你注册的 webhook endpoint 发送实时 webhook 事件通知。把这些通知按 return code 映射到邮件模板,可自动向员工和雇主发送有针对性的邮件。

Payment return lifecycles · 付款退回的生命周期

Option 1: Two day processing · 方案一:两天处理

雇主可在发薪日(payday)前两天发起 funding:

  1. 薪资运行确认:确认后资金转移开始。
  2. 发起 Employer Funding:发薪日前两天,从雇主账户提取资金。
  3. 发起 Payouts:发薪日前一天,将资金推送到员工账户。
  4. Employer Funding Return(雇主付款退回):遇失败的 funding 交易时,雇主账户自动置于 RESTRICTED,阻止发起额外薪资运行,直到问题解决。同时发送 Notification.Payment.employerFundingFailed。
  5. Re-Debit Attempt(重新扣款尝试):Salsa 发起重新扣款,同时发送 Notification.Payment.employerFundingRescheduled。
    1. 成功:发送 Notification.Payment.employerFundingCleared,账户恢复 ACTIVE,可恢复薪资运行。
    2. 失败:自动重试最多三次;若都未清算成功,建议立即让客户体验团队介入。视合同条款,由你的组织或 Salsa 的 Customer Success 团队联系雇主商讨还款计划。

Option 2: Four day processing · 方案二:四天处理

雇主可在发薪日前四天发起 funding:发薪日前四天发起 Employer Funding;遇失败则账户置 RESTRICTED 并发 employerFundingFailed;发薪日前一天发起 Payouts(此时 funding 应已结算 settled),Salsa 发起重新扣款并发 employerFundingRescheduled。

Salsa originated payments · 由 Salsa 发起的付款

对源自 Salsa 的银行交易所使用的模式与实践的说明。

为确保由 Salsa 发起的付款保持一致且易于识别,我们对所有生成的银行交易应用一套简单模式。这带来可预测性,可帮助:入站客户支持咨询(识别付款及用途)、为雇主自动对账/生成分类账(ledger)。

交易总体构成包含以下一个或多个组成部分:

  • Transaction Code(交易代码)——由 Salsa 定义、唯一标识某种付款类型(3 个字符)。
  • Partner Code(合作伙伴代码)——5 个字母的 slug,合作伙伴名称的简短缩写(最多 5 字符)。
  • Partner Name(合作伙伴名称)——合作伙伴名称(最多 16 字符)。
  • Employer Name(雇主名称)——雇主商业名称(最多 16 字符)。

Payroll Elements · 薪资要素

Salsa 在所有薪资要素(payroll elements)上采用一套统一模式,让你能轻松配置并完全掌控薪资处理方式。

在 Salsa,我们通过三个各自独立又协同运作的概念来管理所有影响薪资运行的数据输入:"Types"(类型)、"Policies"(政策)和 "Entries"(条目)。这套模式在所有薪资要素上一致。

Payroll Elements(薪资要素)

薪资要素是指任何工人级别(worker-level)的数据输入,雇主把它带入一次薪资运行,经计算后最终在工人收入记录(earnings record)上生成一条 pay、deduction 和/或 contribution 条目。Salsa 支持:

要素说明
Pay工资与薪酬类型(hourly 时薪、salary、commission 提成、bonus 奖金、reimbursement 报销)
Benefits与福利计划相关的雇主缴款,和/或雇员发起的扣减(401k、health insurance、FSA)
Deductions从税前总薪酬(gross pay)中的扣减(loan repayments、cash advances、union dues)
Garnishments依法院命令或法律从工资中作出的扣减(child support、wage levies、tax liens)
Time Off已累积或可用的休假(PTO、vacation、sick time)
Premiums基于工作条件的特殊薪酬加项(split shift premium、spread of hours premium)
Workers Compensation Insurance用于雇主保险分类与参保管理的工伤赔偿跟踪

每个要素会使用下述 Types、Policies、Entries 的某种组合。本指南示例聚焦 Pay,但概念适用所有要素。并非每个要素都用到每一层级。

Types(类型)

所有要素的 types 构建在 Jurisdictional Types(辖区类型)之上——由合规规则定义的类别,承载某辖区的税务处理(tax treatment)和监管规则。由 Salsa 管理、不可直接配置,是所有 partner/employer types 继承的基础。

一名雇员在单个 pay period 内拥有多种 pay 类型很常见,如 Hourly Pay、Salary、Commission、Tips、Bonus。我们通过 Type 模型捕获,可在 Partner(合作方)层或 Employer(雇主)层收集。

Partner Types(合作方类型)

Salsa 允许你作为 partner 掌控哪些 types 可供 employers 使用。Partner types 是系统级配置。作为账户搭建的一部分,我们为你收集配置所需 types;配置后即可供你的 employers 使用。可添加数量不限、随时添加——新增一个 type 只需点击几下,所有 employers 自动获得使用权限。可通过 Retrieve all PartnerPayTypes API 查看。

📘 示例

你发现所有 employers 通常支付 "Hourly Pay" 和 "Tips",可在 partner 层配置好让所有 employers 立即可用。后续如需 "Cash Tips"(税务处理不同于 Tips)或 "Bonus",也可配置后立即启用。

Reference Id(引用 Id)

每个 type 配置一个唯一 referenceId,在 Sandbox 和 Production 环境间保持一致,这样构建集成时可把 type 当作一个"概念"看待,不必关心每个环境的具体实例。

Employer Types(雇主类型)

另有一层可选自定义:每个 partner type 可进一步配置成针对每个 employer 需求的分支变体。可通过 Retrieve all EmployerPayTypes 管理。

📘 示例

你支持 "Hourly Pay",但 Employer A 想把工时分为 "Shop Hours" 和 "Storeroom Hours",分开显示、各配默认值。可基于 "Hourly Pay" partner type 配置成新的 employer pay types。

Policies(政策)

types 搭建好后,让 employers 配置控制某要素如何为工人处理的循环值和输入,即 Employer Policies 和 Worker Policies。一个 policy 可引用 partner type 或 employer type。

Employer Policies(雇主政策)

让 employer 配置公司级默认值,搭建工人时可继承。通过 Employer profile UI 组件管理。

📘 示例

"Shop Hours" 下 till workers(收银)通常 $12.50/时,floor workers(卖场)$9/时。可配两个 Employer Pay Policy:显示名 "Till Workers" 默认 $12.50、"Floor Workers" 默认 $9,免去逐人设置。

Worker Policies(工人政策)

把一名工人分配到某特定 type 的手段。创建后该 type 开始出现在该工人的薪资运行中。采用 read-through(读透)模式:工人未覆盖某字段时,值从 employer policy 继承。通过 Worker profile UI 组件管理。

📘 示例

Worker A 是 "Till Worker" 但实际 $14/时。可基于 "Till Worker" employer policy 为其创建 worker policy(把 A 附加到 "Shop Hours" 上),再把默认 $12.50 覆盖为 $14,只影响 A。

Worker Entries(工人条目)

代表某 type 的一个具体实例——工人已挣得的那一笔。通过 Paystream 捕获(如需)。

📘 示例

你的系统管理 "Hourly Pay"。账户搭建时禁用该 pay type 使其不再对 employers 可用(你的系统是事实来源 source of truth)。Worker A 于 2024-01-01 以 $14/时工作 8 小时,你通过 Paystream 把它作为 Worker Pay Entry 发来,存入 Salsa 并出现在薪资运行和收入记录上。

❗️ 注意

Worker Entries 优先级高于 Worker Policies。若你既有 "Hourly Pay" worker policy 又发来 "Hourly Pay" worker entry,则薪资运行中只处理这个 entry。

Concepts 概念

Embedded Payroll · 内嵌式薪资

什么是 embedded payroll?

构建、提供并支持一款合规的 payroll 产品很复杂。Embedded payroll 通过提供所需基础设施,让在既有产品内部推出并提供 payroll 变得简单得多。通过处理税务、资金流转、合规等复杂细节,embedded payroll 提供方使产品公司能向客户提供 payroll 服务。但并非所有提供方都一样。

为什么选择 Salsa?

Salsa 提供市场上最现代化的 payroll 基础设施。我们借助数十年 payroll 专业经验,从零重新构想 payroll,而非复制已有的东西。我们的产品设置更快、维护更容易、更灵活、更可扩展、更不易出错,让各类组织能更快推出 payroll,把精力集中在以毫不费力的体验取悦客户上。

Payroll Fundamentals · 薪资基础

确立 Salsa 美国薪资基础的通用术语与定义。本章节描述与美国境内员工薪酬和税款的计算、处理及管理相关的基础概念与实践。

该父页仅含上述引言;实质内容在其下的 7 个子页(内嵌式薪资、雇主信息、员工分类、金额计算策略、员工付款记录、资金流转、员工薪资组)。

Employer Information · 雇主信息

为了准确计算和管理 payroll,Salsa 需要某些雇主信息。一个 Employer 代表雇用并支付 Worker 报酬的法律实体,必须向税务机构登记注册,授权 payroll 服务商代表其申报和缴税,并遵守严格的 KYB 合规要求。

Employer name · 雇主名称

雇主的法定名称,出现在官方文件和申报材料上,应与 IRS 存档名称完全一致。

Work location · 工作地点

worker 被要求履行职责的实际物理地址,用于准确计算和申报税款;一个雇主列出多个 work location 很常见。

Mailing address · 邮寄地址

必需,因为某些州立法规定它必须显示在 worker 付款记录文件上。可与某个 work location 或 filing address 相同,也可单独。

Filing address · 申报地址

也称 legal address(法定地址),是主要营业地点,应与 IRS 存档地址一致;Salsa 代表 Employer 申报税款时使用。

Bank account information · 银行账户信息

必须提供以完成 payroll 付款,含 account name、routing number、account number,并确保账户有效且活动。

IRS Federal EIN · 联邦 EIN

通常称 FEIN,IRS 分配的 9 位数字 ID,使 Salsa 能准确申报缴纳联邦税,应与官方文件一致。

State EIN · 州 EIN

也称 state account ID,由对应州级机构分配,使 Salsa 能准确申报缴纳州税;格式因 tax agency 而异,常见于机构直邮件上。

Deposit frequency · 缴款频率

也称 deposit schedule,由对应州税务机构分配,决定税款到期缴纳频率,常见 semiweekly(每半周)、monthly、quarterly、annual。

Tax rate · 税率

由对应 tax agency 分配,决定雇主需代表 worker 缴税的百分比。

Worker Classification · 员工分类

一个 Worker 会被分类为两种之一:"employee"(员工,用 W-2 表格申报)或 "independent contractor"(独立承包商,用 1099-NEC 表格申报)。

🚧 错误分类可能导致罚款

正确分类 Worker 很重要,这样才能收集正确的设置信息、准确计算税务缴款,并避免因分类错误产生的罚款。

Employee · 员工

有时称 W-2 worker,是为雇主提供服务、且雇主能控制"做什么"和"怎么做"的劳动者。支付给 employee 的款项在到手工资(net pay 净工资)之前需先扣缴税款和各项 deductions。分两类:

  • Exempt(豁免):无权获得加班费,通常是行政管理类、事务管理类、专业技术类及外勤销售类员工。
  • Non-exempt(非豁免):依《公平劳动标准法》(FLSA)享有最低工资和加班费等权利与保护。

Independent contractor · 独立承包商

也称 1099 worker,付款方只有权控制或指导工作"结果",无权控制"做什么/怎么做"。通常不从报酬预扣所得税,每年收到 1099-NEC 表格申报所收报酬,其收入需缴 self-employment tax(自雇税)。

Amount strategies · 金额计算策略

雇主可用几种不同方法计算员工收入。在 Salsa 我们称之为 amount strategies,包括 "fixed amount"(固定金额)、"percentage"(百分比)和 "rate based"(基于费率)。每种都提供一种按每个 pay period 计算收入的方式。

Fixed amount · 固定金额

基于单个 pay period 内设定好的固定美元金额计算报酬,与工时、销售额无关。常见于 tips 小费、bonuses 奖金、flat commissions 固定佣金、salaries 薪水。

Percentage · 百分比

基于某个总额(gross amount)的一个百分比计算报酬。典型例子是佣金(commission):按某总额(如达成销售额)的百分比算。

Rate based · 基于费率

基于每个时间段设定好的 pay rate 计算报酬。最常见是"每小时费率"(按小时计薪),也有"每周费率""每月费率"。

📘 示例

某员工 $10/时,一周工作 30 小时,总收入 $300($10 × 30)。

此 Concepts 页只用叙述性名称(Fixed amount / Percentage / Rate based),API 层可能对应 FIXED/RATE_BASED 等枚举字段名。

Worker payment records · 员工付款记录

提供关于员工每个发薪周期收入、员工代扣项、雇主缴款以及净得工资的重要信息。也常称 "payslip"(工资单)、"paystub"(工资条)或 "paycheck"(薪水支票)。

趣味事实:向员工提供付款记录其实并非联邦法律强制要求,但很多州都有强制规定。

Gross wages · 税前总工资

员工在任何税款、福利或其他扣款从工资中扣除之前的收入总额。

Deductions · 扣款

Pre-tax deductions · 税前扣款

在税款代扣之前从 gross wages 中减去,会降低整体税务负担。常见:401(k)、Medical benefits、Group term life insurance。

Post-tax deductions · 税后扣款

在税款代扣之后从 gross wages 中减去,不降低税务负担。常见:Roth IRA、Union dues(工会会费)、Wage garnishments(工资扣发)。

Employer contributions · 雇主缴款

雇主为雇主发起的计划(如 health insurance 或 401(k))所做的财务缴款,不影响 gross wages 总额。

Reimbursements · 报销

偿付员工自付支出,通常包含在 payroll 中,是不计税的(non-taxable)。

Tax withholdings · 税款代扣

员工和雇主都需就收入缴税。IRS 和各税务机构要求特定代扣,含失业税和所得税。

Employee taxes 员工税:Federal income tax、Social security、Medicare、State income tax(适用时)等。

Employer taxes 雇主税:Social security、Medicare、Federal Unemployment、State Unemployment 等。

Net pay · 净得工资

员工到手工资(take home pay),即所有税款和扣款扣除后的金额。

Net Pay = Gross wages − Employee taxes − Employee deductions

Total payroll cost · 薪资发放总成本

每次处理 payroll 都伴随薪资成本,是一项雇主成本。

Total employer cost = Gross wages + Employer taxes + Employer contributions + Reimbursements

Money Movement · 资金流转

处理 payroll 时有若干步骤需要资金流转,包括 employer debits(雇主扣款)、worker payments(员工付款)、tax payments(税款缴纳)等。

Employer debits · 雇主账户扣款

代表为覆盖员工付款、税款缴纳及报销而需从雇主账户提取的总金额。

🤖 Total employer debit(雇主账户扣款总额)

Gross wages(税前工资总额)+ Employer taxes(雇主税)+ Reimbursements(报销)

🚧 Employer contributions 目前不含在 employer debit 内

目前雇主向公司发起的计划所做缴款会在 Salsa 中被追踪,但需在系统之外与相应 providers 对账核销。

Worker payments · 员工付款

代表员工应收到的已挣得薪酬,通常以 direct deposit(直接存款)、paper check(纸质支票),有时以 cash(现金)发放,由雇主决定方式。

  • Direct deposit——从雇主银行账户自动划转直接进入员工银行账户。
  • Paper check——用以支付已挣工资的实体纸质支票。
  • Cash——用以支付已挣工资的现金付款。

Tax payments · 税款缴纳

代表必须缴纳的各类税款(federal/state/local income tax、social security and Medicare taxes 等)。缴税与报税可能复杂难懂,Salsa 会代表雇主 withholding(代扣)并 submitting(提交)所有 payroll tax payments,确保始终合规。

Worker pay groups · 员工薪资组

一个 worker pay group 是把具有共同特征的员工归为一组的便捷方式,通过满足共同处理要求(pay schedule 发薪日程、worker classification 员工分类、所支付薪酬类型等)来简化 payroll 处理。把一个雇主下的一组员工合并,可高效地用于填充(populate)payroll run(发薪批次)。

Pay Schedule · 发薪日程

最常用的 pay group 是那些同时定义了 schedule 的——常直接称 "pay schedules"。雇主可配置 frequency(发薪频率)及每周/每月具体日期,由此决定 pay period(发薪周期)和 pay date(发薪日)。基于 schedule,pay group 会有一组已知 pay period,便于按循环日程给这组员工发薪。

Webhooks · 事件回调

用 Salsa 的 REST API 测试和管理你的 Webhook 端点。Salsa 用 Webhooks 通知你账户中发生的变化。

创建一个 Webhook 端点

向 webhook-endpoints 发 POST 请求。endpointUrl 是 Salsa 发送事件的目标地址。

curl --request POST --url https://api.sandbox.salsa.dev/api/rest/v1/webhook-endpoints \
  --header 'authorization: Bearer [your-auth-token]' \
  --data '{ "description": "...", "endpointUrl": "https://my-webhook-url.com/path" }'

创建时 status 默认 ENABLED,响应含 signatureSecret(签名密钥)。

更新一个 Webhook 端点

向 webhook-endpoints/{id} 发 PUT,可改 endpointUrl、description、status。禁用后 Salsa 不再向其发事件。

保护你的 Webhook

每次调用含三个 header:

  • Webhook-Id:消息唯一标识(重发时用同一 ID)。
  • Webhook-Timestamp:自纪元起的秒数时间戳。
  • Webhook-Signature:空格分隔、Base64 编码的签名列表。

对内容签名:用 . 拼接 Webhook ID、timestamp、body:signed_content = "${webhook_id}.${webhook_timestamp}.${body}"。验证前不要改动 body。

Salsa 用 HMAC + SHA-256 签名。用签名密钥中 whsec_ 之后的 base64 部分对 signed_content 做 HMAC,结果应与 Webhook-Signature 之一匹配。验证前移除版本前缀(如 v1,)。

🚧 比较签名时用常数时间字符串比较(constant-time),防时序攻击。

验证时间戳:把 Webhook-Timestamp 与你系统时间比较,建议容忍 10 分钟窗口。

Webhook 事件

常见触发动作:Employer/Worker 被创建或更新;onboarding 状态变化;雇主被提醒跑 payroll;Payroll Run 等待输入;Payroll Run 被确认;worker 被发薪。Sandbox 与生产行为一致。

🚧 对 Direct Deposit 发薪的 worker,Notification.workerPayday 不在 Sandbox 发送;可用 Paper Check 方式测试。

重试计划(指数退避)

失败后按:立即 → 5 秒 → 5 分钟 → 30 分钟 → 2 小时 → 5 小时 → 10 小时 → 再加 10 小时。端点被禁用则投递也被禁用。

Filtering and sorting · 筛选与排序

Salsa 让你轻松对请求筛选、对响应排序。通过 filter 参数应用,需提供 fieldName 和值。

按单个值筛选

GET /api/endpoint?filter[stringField]=value

按多个值筛选(数组)

GET /api/endpoint?filter[stringField]=[value1, value2]

按日期筛选

  • onOrBefore(date):该日期当天或之前。
  • onOrAfter(date):该日期当天或之后。
  • onOrBetween(startDate, endDate):落在区间内。

日期区间字段(含 startDate/endDate)用 contains(date)。

应用多个筛选器

GET /api/endpoint?filter[field1]=value1&filter[field2]=value2

排序

指定字段名 + 方向。字段名前加 - 表降序,否则升序。示例:sort=-payDate。支持多字段排序。未指定排序时列表顺序不保证。

字段名规范

  • 与 REST 响应对象字段名对应;区分大小写。
  • 嵌套字段用点号,如 payPeriod.payDate。

Employer Readiness · 雇主就绪状态

了解如何跟踪雇主向员工和承包商发薪的就绪状态。雇主能发薪前必须过两道相互独立的关卡:

关卡它回答的问题
Employer and Pay Worker ReadinessSalsa 是否已拥有计算薪资所需的一切?
Employer Account Standing该雇主是否合法合规、已通过审查可运营?

二者独立:readiness 为 READY 但 standing 为 RESTRICTED 仍不能跑薪,反之亦然。本页讲 Pay Worker Readiness;合规关卡见 Employer Account Standing。

如何跟踪

可从 Salsa Dashboard、Employer Capabilities API、或 webhook 事件 Employer.PayWorkersReadiness.statusChanged 获取。

Statuses · 状态

Status说明
ONBOARDING雇主创建后的初始状态,用托管/嵌入式入驻收集雇主信息并邀请工人;完成后评估是否需 implementation。
IMPLEMENTATION完成入驻并申报此前薪资总额后进入;Salsa 主动对接协助导入。Sandbox 会自动跳过直接到 READY。
READY所有要求满足,可跑薪。
PARTIALLY_READY因配置变化不再满足全部就绪要求(如用工类型改变需带排期的 pay group,或税务设置不完整);可进行有限操作,补齐后回到 READY。

Worker Payment Readiness · 员工可发薪就绪状态

了解如何追踪一名员工从雇主处接收薪资的就绪程度。发薪前必须先完成员工 onboarding(收集个人和雇佣信息)。可通过 Worker Payments Readiness 字段监控。

如何追踪

从 Salsa Dashboard、Worker Capabilities API、或 webhook 事件 Worker.PaymentReadiness.statusChanged 获取。

States · 状态

  • ONBOARDING:员工创建后的初始状态,直到满足至少一个辖区的付款要求。雇主须填雇佣数据(含 classification 和加班资格),税务/银行等可由雇主录入或由员工自助入驻收集。
  • READY:在所分配工作地点付款所需数据齐全,可通过 run payroll 获薪。
  • NOT_READY:此前已入驻的员工因辖区要求变化需补新数据(不常见),补齐后回到 READY。
  • PARTIALLY_READY:在至少一个辖区可付、其他辖区尚不可(多地点分配时),小型企业中较罕见。

Early wage access · 提前支取工资

用 Paystream 和 Worker Payment API 高效管理按需支付和提前支付给工人的工资。Salsa 允许在 payroll run 之外随时向工人发薪。本指南教你创建 payment、在轮班结束时发薪(提前支取)、确认已支付、并获取支付历史。假设你已设好 employer 和 workers。

Create a worker payment · 创建一笔 worker payment

为特定工人创建 worker payment,提供 pay period 和常规 pay date。Salsa 用该工人的 pay schedule(此例为 weekly)计算税款。

curl --location 'https://api.sandbox.salsa.dev/api/rest/v1/employers/${employerId}/workers/${workerId}/worker-payments' \
--header 'Authorization: Bearer ${YOUR_API_TOKEN}' \
--data '{ "payPeriod": { "startDate": "2023-05-01", "endDate": "2023-05-07", "payDate": "2023-05-07" } }'

返回一个 status: "PENDING" 的 worker payment,含 id、初始为 0 的 grossPay/netPay、以及预置的 taxDeductions 和 taxContributions 列表(Social Security、Medicare、Federal Income Tax、CA SDI、CA PIT、FUTA、CA SUTA 等)。

Add compensations · 添加报酬

工人轮班结束后,把当天报酬通过 Paystream 发给 Salsa。Salsa 自动调整上一步的 worker payment 并计算 net pay。

curl --request POST --url 'https://api.sandbox.salsa.dev/api/rest/v1/paystream' \
--header 'X-Salsa-Organization-Id: :partnerId' --header 'Authorization: Bearer ${YOUR_API_TOKEN}' \
--data-raw '{ "data": [ { "compensationReferenceId": "ref:comp:partnername:hourly",
  "startDate": "2023-07-07", "endDate": "2023-07-07", "type": "PaystreamCompensationItemInput",
  "workerId": ":workerId", "hours": "20", "rate": "50", "unit": "HOURS" } ] }'

随后可 GET worker payment,看到本轮班算出的 net pay(例:grossPay 1000.00、netPay 914.50)。

Record the advanced payment · 记录已提前支付的款项

向工人发款后记录这笔提前支付,它会在 pay period 结束时从 net pay 扣除。调 .../advance-payment。

📘 Salsa 假定未结清 net pay 已付给工人,并创建一笔 post-tax deduction 表示已发总额。

Confirm the payment · 确认支付

pay period 结束时确认工人已被发薪:调 .../confirm,返回 status: "CONFIRMED"。

Retrieve payment records · 获取支付记录

可获取该周期的 worker payment records,含 compensations、deductions、tax contributions 和 totals(调 .../worker-payment-record)。

说明:本页含多段结构相近的示例 JSON 响应(worker payment 对象),此处保留关键字段说明,完整逐条 JSON 见 Salsa 原文档 Early wage access 页。

Employer verified bank accounts · 雇主已验证的银行账户

集成你现有的银行账户验证流程,简化雇主 payroll 入驻。若你的应用已在收集并验证雇主银行账户,可通过 Paystream API 同时同步和验证,避免让雇主重复提供信息。

工作原理

  • Payment authorization(支付授权):雇主授权 Salsa 从其账户扣款(debit)。默认在 onboarding 中收集。
  • Account verification(账户验证):Salsa 发起两笔小额存款(micro-deposits),雇主 1-2 天后在 Salsa 输入确切金额以"激活"账户;金额匹配即完成。默认由 Salsa 管理。

🚧 本指南面向已有验证系统的组织。若没有,可用 onboarding 组件收集、验证并授权银行账户。

步骤

1. 内嵌雇主支付授权:用 Paystream API upsert 雇主记录,复用现有授权。

{ "data": [ { "type": "PaystreamEmployerUpsertInput", "id": "er_12345",
  "bankAccount": { "id": "erbank_124345", "paymentAuthorization": { "acceptedAt": "2023-10-16T14:05:22Z" } } } ] }

2. 内嵌雇主银行账户验证:同样 upsert,复用现有验证(需联系 Salsa 启用"绕过银行账户验证")。

"bankAccount": { "id": "erbank_124345", "verification": { "organizationVerifiedAt": "2023-10-16T14:05:22Z" } }

3.(可选)一次请求同时内嵌授权 + 验证,把 paymentAuthorization 和 verification 一起传。

📘 使用此功能时,可考虑在雇主资料和 onboarding 元素中隐藏银行账户卡片。Salsa 可能要求你提供授权/验证的记录。

Set up and pay Overtime · 设置并发放加班费

Overtime(加班)指员工在常规工时之外额外工作的时长,通常在一个 workweek 内,往往按更高费率支付。

谁有资格?

一般 non-exempt(非豁免)员工有资格;exempt(豁免,通常为满足 FLSA 条件的 salaried 岗位)无权获加班费。具体岗位、雇佣协议、州法规都可能影响。

如何计算?

按员工在某 workweek 内、单个 workday 内或连续工作日的工时计算,规则取决于工作地点所在州。通常一周超 40 小时产生 overtime。标准加班费率通常是常规时薪的 1.5 倍;有些州不同,如 California 有 1.5 倍 Overtime 和 2 倍 Double Overtime。

如何设置自动加班计算?

取决于:

1)该 worker 的 WorkerContract 为 EMPLOYEE 分类且 overtimeEligible: true。

2)把按天拆分的 hours 通过 Paystream Payroll Elements 端点发来。

🚧 Salsa 计算 overtime 要求粒度必须逐日(day by day);若某 worker 存在跨越多天的 pay entry,则不为其算 overtime。

--data-raw '{ "data": { "type": "PaystreamPayrollElementPeriodReplacementInput",
  "workers": [ { "pay": [
    { "hours": 20, "startDate": "2024-01-01", "endDate": "2024-01-01", "rate": "15.00", "payReferenceId": "${HOURLY_REF_ID}" },
    { "hours": 8,  "startDate": "2024-01-02", "endDate": "2024-01-02", "rate": "15.00", "payReferenceId": "${HOURLY_REF_ID}" }
  ], "workerId": "${WORKER_ID}" } ],
  "periodStartDate": "2024-01-07", "periodEndDate": "2024-01-01" } }'

3)该 worker 须在某 workweek/单日/连续日工作达到一定小时数(取决于州规则)。

📘 California 示例:一周超 40 小时按 1.5×;单日 8–12 小时按 1.5×、超 12 小时按 2×;一周第 7 个连续工作日前 8 小时 1.5×、超出部分 2×。

4)完成后,每次 Payroll Run 期间自动在后台计算 overtime,并显示在工资单上;两种加班金额都可在 Payroll Run 中手动覆盖。

📘 示例:CA 员工、逐日 Hourly、周排班,周一 13 小时、周二 6、周三 6,时薪 $3,共 25 小时 → 20 小时 Hourly($3)、4 小时 Overtime($4.5)、1 小时 Double Overtime($6)。

🚧 若你希望 Salsa 不自动算 overtime,可启用 Overtime pay types,由你直接发金额。

用 Usage API 检索 Salsa 平台上各 employer 的用量指标(usage metrics),帮助你向客户计费。可按日期范围检索某个或所有 employer 的指标。

指标说明

WORKER_PAYMENT已完成的 worker payment 数量。
WORKER_PAID获支付的 worker 去重数量。
WORKER_ANNUAL_TAX_FILING_SENT已发送的 worker 税务申报数量。

Historical Workers Workflow · 历史员工数据流程(合作方集成指南)

客户从此前服务商迁移时,须一并纳入历史员工(在迁移前已被终止或非活跃的员工),以保证数据完整和 payroll 记录准确。

什么是历史员工?

出现在客户历史 payroll 数据(过往 pay run 或税务文件)中、但接入 Salsa 时已不在职的员工。常见缺失原因:注册前已终止;或角色超出你平台业务范围(如发廊的维修承包商)。因此他们可能不在你的员工目录中、缺 externalId、遗漏 W2/1099 等文件。

Salsa 的处理方式

为新雇主 onboarding 时,Salsa 实施专员可能手动创建历史员工以回填 payroll 历史。创建时触发常规 Worker.created webhook,但注意:terminationDate、terminationCategory: HISTORICAL、且 workerExternalId 应不存在(表明在 Salsa 内部创建)。

External IDs & Mapping to Salsa · 外部 ID 与到 Salsa 的映射

用 External ID(外部 ID)在你系统与 Salsa 之间可靠映射和同步实体,实现持久引用、幂等创建、更简便的 webhook 处理。

什么是外部 ID?

特定 Salsa 实体(Employer、Worker、Address、Bank Account 等)上的一个字符串字段,特点:可选(默认空)、在该实体类型命名空间内唯一、由你(合作方)提供并拥有、在 API/webhook/工作流中被当作持久的一等公民数据。

为什么用?

  • 可靠的实体映射:把你系统标识符直接嵌入 Salsa。
  • 更好的同步与 webhook 处理:外部 ID 出现在响应和 webhook 载荷中。
  • 幂等创建与防重复:创建时传外部 ID,若已存在返回唯一性约束错误,防重复。

命名空间与唯一性

唯一性按实体类型全局强制。即使两个 Worker 属于不同 Employer,其 externalId 也须在所有 Worker 中唯一。若你的 ID 仅局部唯一(如作用域在某 Employer 内的 Worker ID),可构造复合 ID 保证全局唯一,如 acct_123:17;也可哈希或生成稳定 GUID。

最佳实践

  • 对任何想与 Salsa 保持稳定 1:1 映射的实体都用外部 ID。
  • 用于 webhook 路由和同步时的幂等创建。
  • 不要当元数据用——应保持为稳定标识符。

集成流程示例

创建 Employer 时指定 externalId(如 acct_123);同时把 Salsa 返回的 Employer ID 也存进你的库。处理 PayrollRun.awaitingInput 等 webhook 时,事件含 employerExternalId,可仅凭它路由,无需额外查找。

Protecting sensitive actions with identity verification · 用身份验证保护敏感操作

在用户于 Salsa 嵌入式 UI 执行敏感操作前,要求一次额外身份检查(如 2FA/MFA)。若你已有身份验证系统,可与 Salsa 嵌入式 UI 集成。典型例子:取消对银行账号或政府证件号的掩码显示(unmasking)。

工作原理

用户尝试其当前角色未被授予的操作时,Salsa 显示验证提示并发出 request-privileged-access 事件(载荷含 action、recommendedRole、possibleRoles)。你的应用跑自己的验证流程,按推荐角色创建 token 交回,Salsa 随即恢复操作。

角色与提权阶梯

提权阶梯:basic → admin → super-admin,每层含下层全部权限再加更多。示例(雇主通道):add-employer/worker-bank-account、update-worker-government-id、download-document 需 admin;unmask 各类(银行账号、政府证件)、delete-worker-bank-account、下载含政府证件的报表需 super-admin。

设置政府证件号是否必填(MANDATORY/APPLIED_FOR)不属特权操作,不触发提权。

实现步骤

1)创建带初始角色层级的 user token(嵌入式建议从 admin 起步)。2)创建 element 时设 allowRequestForPrivilegedAccess: true。3)监听 request-privileged-access,跑验证后按 event.recommendedRole 建 token 并 element.replaceUserToken(token)。4)短时间(建议 5–15 分钟)后主动降级到更低角色。

element.on("request-privileged-access", async (event) => {
  const verified = await runYourVerificationFlow();
  if (!verified) { element.cancelRequestForPrivilegedAccess(); return; }
  const token = await createToken({ role: event.recommendedRole });
  element.replaceUserToken(token);
});

recommendedRole/possibleRoles 均可选;用户无提权路径时 Salsa 会省略,建 token 前须检查其是否存在。

🚧 禁用的控件仅用于显示,限制在 API 层强制:缺少所需角色的请求总会被拒绝。设 allowRequestForPrivilegedAccess: false 则无任何提权路径(默认)。

Set up accounting integration · 设置会计(记账)集成

从 Salsa 拉取工资单会计凭证(payroll journal entries)并导出到你的会计系统。跑完薪资后:1)拉取凭证(借/贷金额);2)在你系统创建凭证;3)标记为已导出。

第 1 步:为雇主启用会计

mutation CreateAccountingConfiguration {
  createAccountingConfiguration(input: {
    employerId: "emp_abc123", provider: MANUAL, workerPaymentExportTrigger: ON_CLEARANCE
  }) { accountingConfiguration { id provider workerPaymentExportTrigger } }
}

导出触发:ON_CLEARANCE(推荐,款项清算后才就绪,更准)/ON_CONFIRM(确认时立即就绪,更快但可能时间错位)。

第 2 步:设置科目映射

把工资单要素映射到你的会计科目表。先查询发现需映射项及 mappingId,再配置科目(用 externalId 存你的科目编码)。科目类型:ASSET(银行/清算账户)、EXPENSE(工资/报销/雇主税缴纳)、LIABILITY(税款代扣/应付扣款)。映射策略:defaultAccounts(全用同一科目)、byPayrollElementType(按要素类型)、byTax(按税种)。

第 3 步:接收并导出凭证

Webhook 事件:Transmission.processing/Transmission.completed/Transmission.failed。收到 completed 后用 transmissionId 拉取凭证(GraphQL),含 journalEntry(transactionDate、totalDebit/Credit、lines)。明细行类别:BANK_ACCOUNT/WAGE/REIMBURSEMENT/TAX_CONTRIBUTION/NON_TAX_DEDUCTION/NON_TAX_CONTRIBUTION。记账方向 DEBIT/CREDIT。导出后调 markTransmissionCompleted。失败可查询 FAILED_TO_TRANSMIT 的需求项并用 createTransmissionForRequirements 重试。

CSV 导出

每份凭证也含 CSV 下载(csvDocument.links.url,有 expiresAt 过期时间)。CSV 字段(RFC 4180):Transaction Date、Account Id/Name/External Id、Category、Posting Type、Amount。

说明:本页含多段 GraphQL mutation/query 示例(建配置、查/配映射、拉凭证、重试),此处保留流程与字段说明;完整 GraphQL 代码见 Salsa 原文档 accounting integration 页。

Accessing Audit Data · 获取审计数据

审计条目(audit entries)记录 Salsa 平台内各实体的变更:是谁、变更了什么、何时。用途:合规(防篡改的变更历史)、变更追踪、调试。

查询审计条目

通过 auditEntries GraphQL 查询访问,分页,默认按时间戳升序。也可在某实体(如 Worker/Employer)上用 auditEntries 字段查限定条目。

分页

基于页码:page(默认 0)、size(默认 500,最大 500)。响应含 totalPages。持续递增 page 直到 totalPages - 1。

过滤器(filterBy)

  • entityType:按实体类型(如 "WorkerContract")。
  • workerId:某特定 worker。
  • auditTimestamp:日期范围(start/end,均含)。

排序

默认按 AUDIT_TIMESTAMP 升序,可改 orderBy: { field: AUDIT_TIMESTAMP, sort: DESCENDING }。

可用字段

id、auditTimestamp(UTC)、entityId、entityType、action(CREATE/UPDATE/DELETE)、event(人类可读描述)、entity(变更后状态的 JSON)、actor(执行者:AuditUser/AuditApiClient/AuditSystem)、employer、worker。

审计覆盖

涵盖资金流转(银行账户、资金流转策略、薪资分配)、档案(税务设置、地址、合同、工作地点、姓名)、薪酬(payroll run 运行/确认/取消/删除、纳入排除工人、薪资要素、合作方薪酬设置)、会计(合作方会计设置)的创建/更新/删除等动作。

理解变更

entity 是变更之后的 JSON 快照(不存 diff、不存旧状态)。要看变更,对同一 entityId 的连续条目快照做比较。event 提供人类可读摘要,便于直接在 UI 展示。

Premium Pay · 加成工资

Premium pay 指满足某些条件时,Salsa 在员工基础工资之上计算并叠加的额外报酬,由各辖区劳动法驱动。当前支持(均限 New York):Split Shifts、Spread of Hours(按天计)、Wage Parity(按小时报酬计)。

Split Shifts & Spread of Hours

New York 工资令下的 premium pay 要求:满足条件时须保证当天总报酬达最低门槛,不足则补差(top-up)。

  • Split Shift(分段班):同一工作日内两段或更多分开时段、中间休息超过一小时。下限=最低工资×工作小时数+1 小时最低工资;实际不足则补差。
  • Spread of Hours(工时跨度):从当日第一个班次开始到最后一个结束的总跨度(含休息/间隔)超过10 小时时,适用同样下限。

不支付的情况:收入已超下限;员工为豁免或承包商;班次分处不同州。每日只算一次规则:同一天同时触发两者,只付导致下限更高的那一项(但两者各自计入 YTD)。

如何通过 Paystream 发送

pay 和 timeWorked 条目接受可选 premiumEligibility 对象(splitShift、spreadOfHours 布尔,默认 false)。标志只是让 Salsa 评估,不保证一定付。仅支持基于班次(有 startDateTime/endDateTime)的条目;pay 条目须基于费率(rate-based)。工作地点须在 New York,否则返回 HTTP 400。

"premiumEligibility": { "splitShift": true, "spreadOfHours": true }

加成会作为独立收入行加入收入记录,不增加已工作小时数、不影响加班计算,计入 YTD。

Wage Parity

New York 法规下确保某些居家护理员工获得的最低总时薪(基础工资+补贴)高于标准最低工资。费率由 NY 卫生部按地区发布,Salsa 保持最新,合作方无需管理。支持地区:New York City;Long Island & Westchester。按班次评估。履行策略(每名员工须配 Worker Premium Policy):Pay(补贴以现金加在基础收入上,计入 gross/taxable)或 Benefits(雇主按固定每小时金额缴至指定福利,记为非税公司缴纳)。政策在 Salsa 雇主门户按生效日期管理。

Set up Wage Parity Integration · 设置工资平价(Wage Parity)集成

通过 Paystream API 发送符合 wage parity 资格的班次并管理工作地点。概念见 Premium Pay 页。

概览

要让 Salsa 评估某班次的 wage parity,须在每条符合资格的 pay 或 timeWorked 条目上发送两样:1)premiumEligibility.wageParity: true;2)一个 workLocationId 标识班次执行地点。该地点须引用一个能解析到受支持 NY 地区的 EmployerExternalWorkSite(或 EmployerAddress)。

"premiumEligibility": { "wageParity": true }
"workLocationId": "worksite-456"

📘 在 pay 条目发 wage parity 时,所引用的 pay type 必须被配置为计入 time worked,否则不计算津贴。提示:无需为每个病人地址建工作地点,费率按地区定,每座城市一个工作地点即可复用。

Employer External Work Sites · 雇主外部工作地点

表示工人执行工作但不归雇主所有的地点(如家庭护理病人的家)。接受部分地址(街道级可选),足够 Salsa 确定 wage parity 地区。通过 Paystream employer upsert 创建/更新。必填字段:city、state(小写,目前仅 ny)、postalCode、countryISO;可选:id、externalId、name、addressLine1/2。

发送 Wage Parity 班次

可在一个请求中同时建工作地点并发班次(POST /api/rest/v1/paystream?synchronous=true,data 数组按顺序处理),或分两个请求(先 upsert 工作地点到 /paystream/employers,再发班次到 /paystream/payroll-elements)。

校验规则(违规返回 HTTP 400)

  • wageParity: true 时 workLocationId 必填;反之带 workLocationId 但无 wageParity 也被拒。
  • workLocationId 必须能解析到唯一一个已存在实体;解析到多个则被拒(用 Salsa ID 或给 external ID 加命名空间)。
  • 地点须在 New York;条目须基于班次;pay 条目须基于费率。

Employer Account Standing · 雇主账户状态

了解 Salsa 如何在雇主运营前对其筛查和验证。Employer Account Standing 是雇主要在 Salsa 运营必须通过的合规与风险关卡,反映 KYB/AML 筛查、制裁名单核查和持续风险审查的结果。

它与 Readiness 并行:Readiness 问"是否有计算薪资所需数据",Account Standing 问"是否被允许运营"。两者独立且都须通过才能跑薪;readiness 为 READY 但 standing 为 RESTRICTED 仍不能发薪。

如何追踪

从 Salsa Dashboard 或 webhook 事件 Employer.AccountStanding.statusChanged 获取。

Statuses · 七种状态

Status说明
NEW_ACCOUNT创建时初始状态,直到 KYB/AML 要求完成。
IN_REVIEW满足新账户要求后进入;雇主与工人都接受筛查。
ACTIVE已获批、无未完成要求,可不受限运营。
CONDITIONAL有条件批准;Salsa 执行额外尽职调查,可能有未完成要求。
RESTRICTED有未完成要求导致无法发薪;可能涉及信息补充请求(Request for Information)。
REJECTED高风险账户,或行业为 Salsa 无法服务的行业。
CANCELLED因雇主订阅取消而进入。

典型流程:NEW_ACCOUNT → IN_REVIEW → 导向 ACTIVE/CONDITIONAL/RESTRICTED/REJECTED;可基于持续风险评估在非终态间转移。

用 Retrieve all OrganizationCompensationPolicies 获取预配的 compensation policies(含 referenceId、calculationMethod 如 FIXED_AMOUNT / HOURLY_BASED_AMOUNT)。

用 POST /paystream 创建 worker pay entries(结构同 Express 教程:固定额 + 逐日 hourly)。

处理薪资(跑 payroll run):POST /employers/${employerId}/payroll-runs,传 payPeriod(startDate/endDate/payDate)、workerIds、type: "PayrollRunCalculateFromWorkerListInput"。返回含 status: "PENDING"、totals、workerPayments(含 grossPay/netPay、compensations、taxDeductions、taxContributions)。保存 payroll run id。

确认 payroll run:POST .../payroll-runs/${payrollRunId}/confirm,返回 status: "CONFIRMED"。资金流转细节见 Money movement。

第 5 步:向 Workers 交付付款记录

付款记录(earning statements / paystubs)是 payroll run 的产出。用 GET .../payroll-runs/${payrollRunId}/worker-payment-records 获取,含所赚金额、预扣税、扣款、totals、YTD 等,并含一个 document ID。

用该 document ID 调 POST /documents/${documentId}/link 生成 PDF 的 Document Link(返回 url 和 expiresAt),可直接交付给 Workers。

🚧 该 URL 无需额外授权即可访问,务必只分享给预期接收者。
